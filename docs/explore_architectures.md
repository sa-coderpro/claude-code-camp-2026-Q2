## Explore Agent Architecture

The largest confusion tech professionals have is applying the correct agent solution because many solutions 
appears to overlap responsibilities.

## 1. An agent file with referenced files eg. Agent.md, @~/docs/*.MD
The Simplest agent is creating an "agent file" and possibly importing other files that are read conditionally when needed.

We should attempt to create an agent file and see if it can connect to the MUD and complete a simple goal:
eg. "Find a bakery and list the menu"

We want to start with the smallest and least intelligent model to explore first and on need basis can scale up the model to explore more efficiently.

## Technical Observations:

Using haiku 4.5 create CLAUDE.md file with a simple prompt, and told it will need to manage its own local memory via simple markdown files. We provided it with the location of the MUD and the player credentials.

The Agent struggled to connect to the MUD.
The Agent would attempt to create temporary code files to manage a telenet connection and execute commands.
The agent did not have enough information about the Text user interface of the MUD to login and see its mistakes.

## Technical Conclusion

A coding harness alone is not a good fit for operating the MUD.

The deterministic login and connection flow should be handled by a
reusable MUD SDK rather than regenerated by the model on every attempt.

A specialized agent loop, potentially using an MCP server connected to
the MUD SDK, would provide better reliability, visibility, and token
efficiency.
